## TEXTO DO POST (copiar e colar no LinkedIn)

---

Eu tenho um problema sério com projetos pessoais.

Começo, fico empolgado, abandono.

Não é falta de capacidade técnica. É falta de estrutura. A ideia surge, jogo no Claude Code ou Codex, vejo o código nascendo na tela — aquela dopamina toda. Aí algumas horas depois tá tudo virado, contexto perdido, sem saber o que fiz, o que falta, quanto gastei. Fecho o terminal e nunca mais volto.

Meu GitHub é um cemitério de ideias que tiveram 3 commits.

Não sei se é a sua realidade. É a minha.

Então resolvi construir uma ferramenta pra isso. Com IA, obviamente.

Nasceu o Vibe Workbook.

É uma aplicação local (Express + WebSocket) que roda terminais PTY reais no navegador — Claude Code, Codex CLI, Gemini CLI lado a lado — orquestrados dentro de um pipeline de produto com etapas reais:

Idea → Brief → Spec → Architecture → Implementation → Test

Cada etapa produz artefatos. O handoff de uma vira contexto da próxima. Quando abro o terminal amanhã, sei exatamente onde parei e por quê.

O que ele faz hoje:

→ Múltiplos produtos em paralelo, cada um com seu próprio pipeline e estado
→ Agentes diferentes rodando lado a lado em terminais reais (não mock, não simulação)
→ Rastreamento de custos por modelo, por sessão, por agente — $72.74 investidos até agora, sei exatamente onde foi cada centavo
→ Knowledge Packs que injetam contexto técnico relevante em cada etapa
→ Histórico completo de sessões com timestamps

O mais concreto: o ZapCam (monitoramento de câmeras IP com alertas via WhatsApp + análise por IA) saiu da fase de "seria legal fazer isso" e hoje tem Brief, Spec, Architecture e Implementation completos. 46 testes passando. Pipeline visível.

Não é mágico. Continua sendo trabalho. Mas agora eu termino o que começo.

Próximos passos: deploy automatizado, estratégia de marketing e monitoramento de produção integrados direto no pipeline. A distância entre "ter uma ideia" e "ter um produto no ar" deveria ser um processo, não uma aposta.

Repo open source — link nos comentários.

## DICAS DE PUBLICAÇÃO

1. **Horário:** Terça a quinta, entre 8h-10h ou 17h-19h (horário de Brasília)
2. **Primeiro comentário:** Cole o link do repositório GitHub no primeiro comentário, não no post. O algoritmo do LinkedIn penaliza links externos no corpo do post.
3. **Engajamento inicial:** Nos primeiros 30-60 minutos, responda todos os comentários. O LinkedIn prioriza posts com conversas ativas.
4. **Hashtags:** Máximo 5-7, coloque no final. As que estão acima já estão calibradas.
5. **Formatação:** O LinkedIn aceita os emojis → como bullet points. Mantenha parágrafos curtos (1-3 linhas). O post acima já está formatado pra mobile.
6. **Alt text:** Adicione descrição nas imagens para acessibilidade (ex: "Dashboard de custos mostrando $72.74 gastos em agentes de IA").
